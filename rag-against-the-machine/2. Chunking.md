
Split each file into retrievable units ≤ 2000 chars. The chunk — not the file — is what gets indexed, scored, and returned. Theory: [[chunking]].

## Structure-aware

- **Python**: `ast.parse` → one chunk per top-level function/class. `lineno`/`end_lineno` are **1-based**; convert to char offsets via a per-file line-start offset table. Span starts at the **first decorator**. On `SyntaxError` → fall back to line-window chunking.
- **Markdown/rst/txt**: split at headers, merge/split sections to the size cap.

## The invariant that silently breaks recall

```python
open(file_path, encoding="utf-8").read()[first:last] == chunk.text
```

Offsets are ground truth. Never rebuild text from split lines — newline handling breaks the round-trip. A pytest asserts this on real corpus files; it must pass before any chunking change merges.

## Size bias

IoU bar is 0.05 → right file + rough region is what counts → bias toward **large chunks (near 2000)**. Big chunk = better span overlap, slightly diluted BM25 term stats — the tradeoff favors big here.
