Once the function name is locked in, the state machine knows which function was selected — and therefore which argument keys and types are valid.

The constraint enforces:
- **Keys**: only argument names defined for the chosen function (`a`, `b` for `fn_add_numbers`)
- **Types**: values must match the schema (`number`, `string`, or `boolean`)
- **Type coercion**: numbers in prompts may appear as integers but must be output as floats per schema (`40` → `40.0`)

The LLM is doing the semantic understanding — it knows "sum of 40 and 2" maps to an addition function with those two values. The constraint layer ensures it can only *express* that understanding as valid, schema-compliant JSON.

**The constraint is on the form, not the meaning. The LLM handles meaning; you handle structure.**
