Encode the full context text into token IDs and feed it to the LLM.

The tokenizer converts the text into a one-dimensional sequence of integer token IDs. Each ID maps to a string in the vocabulary (a word, subword, punctuation, or special symbol).

This is handled via the vocabulary JSON file — the same file used by `vocabulary.py` for prefix matching during constrained decoding. Every token ID has a known string representation, which is what makes constraint checking possible at inference time.

Important: function names like `fn_add_numbers` may be split across multiple token IDs by BPE. This is why constraint checking requires prefix matching, not exact token lookup.
